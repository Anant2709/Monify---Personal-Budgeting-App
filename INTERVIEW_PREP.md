## Feature-by-Feature Walkthrough

### 1. Spending Dashboard
- **What it does**: Shows account balance, monthly spending vs income, savings rate, spending by category (pie chart with month filter), monthly trends (area chart), top merchants, and a fully filterable transaction table.
- **How it works**: Frontend makes 4 parallel API calls on load (summary, transactions, category breakdown, monthly trend). Backend uses SQLAlchemy ORM with JOIN queries across three normalized tables (transactions, merchants, categories).
- **AI used for development**: Cursor helped generate Recharts configuration and TailwindCSS styling.
- **What I modified from AI suggestions**: Normalized the database from a flat table into three tables with foreign keys. Added month-filter for the pie chart (my own idea). Built the category/date-range filter bar for the transactions table.

### 2. Add Transaction (Manual Entry)
- **What it does**: Prominent "+ Add Transaction" button on the dashboard opens a modal with expense/income toggle, merchant, amount, category dropdown, date picker, and description. Dashboard auto-refreshes after adding.
- **How it works**: Frontend modal calls `POST /api/transactions` with form data. Backend finds-or-creates the merchant and category, then inserts the transaction. The entire dashboard re-fetches to reflect the change.
- **Key UX decision**: Made this the most visible action on the dashboard (top-right blue button) since adding transactions is the core user action for any finance app.

### 3. Transaction Filters
- **What it does**: Search by merchant, filter by category dropdown, and date range picker with calendar inputs. Clickable category badges in the table for quick filtering. Shows filtered net total and "Clear filters" action.
- **How it works**: All filtering is client-side on the already-loaded dataset (365 days). `useMemo` recomputes the filtered list whenever any filter changes, keeping it performant.
- **Key decision**: Client-side filtering avoids extra API calls and gives instant feedback. The 365-day window ensures receipt scans with older dates are still visible.

### 4. AI Spending Insights
- **What it does**: Displays 4 AI-generated insight cards on the dashboard (e.g., "Dining spending up 35%", "You could save $50/month by reviewing subscriptions").
- **How it works**: Backend collects last 30 days of transactions, compares with the previous 30 days, sends both to OpenAI with a structured prompt asking for exactly 4 insights in JSON format.
- **Key decision**: I ask the LLM to return structured JSON rather than free text — this gives me control over the UI layout (card type, icon, severity) while letting AI handle the analysis. Includes in-memory caching (2-minute TTL) to prevent redundant API calls.

### 5. AI Financial Advisor Chat
- **What it does**: A conversational interface where you can ask about budgeting, 401(k), Roth IRA, investing — and it gives personalized answers based on YOUR actual spending data. Responses render with proper markdown formatting (headers, bold, lists, emojis).
- **How it works**: System prompt defines a financial advisor persona with US finance expertise. Your last 30 days of spending (categories, totals, top merchants) is dynamically injected into the system prompt. Responses stream via Server-Sent Events (SSE) for real-time feel. Chunks are JSON-encoded to preserve newline characters through SSE transport.
- **What I changed**: (1) Cursor suggested WebSocket — I switched to SSE because it's simpler and unidirectional. (2) Discovered newlines were being stripped during SSE streaming — fixed by JSON-encoding each chunk on the backend and decoding on the frontend. (3) Added formatting instructions to the system prompt so the LLM responds with proper markdown structure, emojis, and spacing.

### 6. Chat Persistence & History
- **What it does**: Conversations persist when navigating away and back. A sidebar lists all past chats (titled by first message). "New Chat" button starts fresh. Delete any chat with a trash icon. All history survives page refreshes.
- **How it works**: Chats are saved to `localStorage` as an array of `{id, messages, updatedAt}`. Every message send triggers a save. The active chat ID is tracked in React state.
- **Key decision**: Used `localStorage` rather than a backend database for chat history because: (1) no user authentication yet, (2) keeps chat data private on the client, (3) zero additional API complexity. In production with auth, this would move to a database.

### 7. Smart Budget Alerts
- **What it does**: Detects spending anomalies and shows them as prioritized alerts (high/medium/low) with severity badges and actionable suggestions. Alerts are sorted by severity — high always appears first.
- **How it works**: Two-stage process — (1) Rule-based anomaly detection in Python (category spikes >30%, large transactions >2x category average, new merchants), then (2) those raw anomalies are sent to OpenAI to generate friendly, human-readable alert messages.
- **Key decision**: Hybrid approach — deterministic rules catch anomalies reliably (AI can hallucinate patterns), AI makes them engaging and readable (rules produce dry data). Frontend sorts by severity for proper prioritization.

### 8. Receipt Scanner
- **What it does**: Upload a receipt photo, AI extracts merchant, date, total, line items, and suggests a spending category. One click to add all line items as individual transactions.
- **How it works**: Image is sent to OpenAI's GPT-4o-mini vision capability as a base64-encoded payload. Prompt asks for structured JSON extraction. Result is displayed for user confirmation. On confirm, each line item becomes a separate Transaction row — all sharing the same merchant and date, but with individual amounts and descriptions.
- **What I changed**: (1) AI originally generated a single-transaction confirm endpoint. I changed it to save each line item as its own transaction for granular analytics. (2) Falls back to saving the total as one transaction if no line items are extracted.

## Anticipated Interviewer Questions

### About the Project
**Q: Why did you choose to build a budgeting app?**
A: "Intuit builds products that help people manage their finances — Mint, Credit Karma, TurboTax, QuickBooks. I wanted to show I understand the domain and can think about real user problems in this space. Financial data is also a great fit for AI because there are clear patterns to detect and personalized advice to give."

**Q: How would this work with real bank data?**
A: "The architecture is designed for it. You'd swap in Plaid's API to pull real transactions — the data model is identical (date, merchant, amount, category). The AI features work on transaction data regardless of source. The main additions would be OAuth for Plaid, a webhook listener for real-time transaction updates, and user authentication."

**Q: What makes this different from Mint or other budgeting apps?**
A: "Three things: (1) The AI advisor chat that knows your actual spending and gives personalized US-specific financial advice — not generic tips. (2) The receipt scanner that extracts individual line items, not just totals. (3) Smart alerts that combine deterministic anomaly detection with AI-generated natural language — more reliable than pure AI, more readable than pure rules."

### About AI Usage
**Q: Where specifically did AI help you build this?**
A: "I used Cursor throughout. It was most helpful for: boilerplate generation (saving 30-40% of typing), chart configuration (Recharts has a complex API), and as a sounding board for architectural decisions. I kept a running log of every AI suggestion I accepted, rejected, or modified."

**Q: What did the AI get wrong?**
A: "Several things: (1) Suggested WebSocket for chat streaming — overkill, SSE is simpler. (2) Generated a denormalized database — I normalized it for analytics. (3) Receipt scanner saved one transaction per receipt — I changed it to per-item. (4) SSE streaming stripped newlines from LLM responses — I had to debug this and add JSON encoding. (5) Some TailwindCSS v4 syntax was wrong."

**Q: How do you decide when to accept AI code vs write it yourself?**
A: "I accept AI output for well-understood patterns — CRUD endpoints, component boilerplate, CSS layouts. I write or heavily modify anything requiring domain understanding — anomaly detection logic, prompt engineering, architectural decisions. My rule: if I couldn't explain it line-by-line to you right now, I shouldn't have accepted it."

### Technical Deep Dives
**Q: Explain the AI insights feature end-to-end.**
A: "Dashboard loads → frontend calls GET /api/insights → backend queries last 30 and previous 30 days of transactions using JOINs across 3 tables → groups by category → formats into a prompt asking for exactly 4 JSON insight objects → sends to OpenAI GPT-4o-mini → parses the JSON response (strips markdown code fences if present) → returns to frontend → renders as typed cards (warning/success/info/tip). There's a 2-minute in-memory cache so repeated loads don't hit the API. If OpenAI fails, a fallback error card with a specific message (rate limit vs API key vs generic) is shown."

**Q: Why SQLite instead of PostgreSQL?**
A: "For a demo, SQLite is zero-config — no server to install, the database is a single file. The SQLAlchemy ORM layer means switching to Postgres is literally a one-line connection string change. For production with multiple users, I'd use Postgres — see stretch goals."

**Q: Why did you normalize the database?**
A: "The AI generated a flat table with merchant and category as plain strings. I refactored into three tables with foreign keys. Benefits: (1) Vendor analytics — 'Which merchant do I visit most?' (2) Data quality — no typo duplication ('Starbucks' vs 'starbucks'). (3) Proper relational design. Tradeoff is JOINs, but SQLAlchemy's `relationship()` makes that ergonomic."

**Q: How does the chat streaming work?**
A: "Frontend POSTs message + history to /api/chat/stream → backend builds system prompt with financial context injected → calls OpenAI streaming API → each chunk is JSON-encoded and sent as an SSE `data:` line → frontend reads with ReadableStream, JSON-decodes each chunk, and appends to the message → ReactMarkdown renders the accumulated text with formatting. The JSON encoding was a bug fix — raw SSE strips newlines, which broke markdown formatting."

**Q: What happens if the OpenAI API is down?**
A: "Each AI feature degrades gracefully. Insights show a fallback error card, chat displays an error in the bubble, alerts show an error alert, scanner shows 'Scan Failed'. The dashboard itself (charts, transactions, filters) works completely without AI — that's all database queries. Error handling specifically detects rate limits vs API key issues vs generic errors to give actionable messages."

**Q: How does chat persistence work?**
A: "All conversations are stored in localStorage as a JSON array of `{id, messages, updatedAt}`. Every message triggers a save. The sidebar derives chat titles from the first user message. This survives navigation and page refreshes. It's client-side only — in production with auth, I'd move this to a database with user-scoped queries."

**Q: Why client-side filtering instead of server-side?**
A: "The transaction table loads 365 days of data upfront (~200-300 rows). At that scale, client-side filtering with `useMemo` is instant and avoids extra API round-trips. If we had thousands of transactions, I'd paginate server-side with query params. It's about choosing the right approach for the data volume."

## Key Tradeoffs & Decisions

1. **Mock data vs. Plaid**: Chose mock data to avoid API complexity in a demo. Architecture supports real data — same schema, just swap the data source.
2. **Normalized DB vs. flat table**: AI generated flat; I normalized for vendor analytics and data integrity. Tradeoff is JOINs, mitigated by ORM relationships.
3. **SSE vs. WebSocket**: SSE for streaming — simpler, unidirectional, natively supported by FastAPI. WebSocket only needed for bidirectional.
4. **OpenAI vs. Gemini**: Started with Gemini (free tier), switched to OpenAI (GPT-4o-mini) for reliability and better rate limits. The `llm_service.py` abstraction made this a clean swap — same function signatures, only internals changed.
5. **Hybrid alerts (rules + AI)**: Rules detect anomalies deterministically, AI generates natural language. More reliable than pure AI, more engaging than pure rules.
6. **JSON structured output from LLM**: Insights and alerts request JSON — predictable UI rendering, easier error handling, cleaner separation of concerns.
7. **Per-item receipt saving**: Each line item becomes its own transaction. More rows but much richer analytics. Graceful fallback to total.
8. **localStorage for chat history**: Appropriate for single-user demo. Would move to database with user auth in production.
9. **Client-side transaction filtering**: Instant UX for ~300 rows. Server-side pagination for production scale.
10. **Axios for REST + native Fetch for streaming**: Axios auto-parses JSON and handles errors; Fetch gives access to ReadableStream for SSE. Right tool for each job.

## Stretch Goals (Future Enhancements)

### 1. Real Bank Account Linking (Plaid API)

- Set up Plaid webhooks for real-time transaction syncing
- Map Plaid's transaction schema to our normalized tables

- Role-based access if expanding to family/shared budgets
- **Why it matters**: Required for any production deployment — currently single-user only

### 3. PostgreSQL + Cloud Deployment
- Swap SQLite for PostgreSQL (one-line SQLAlchemy connection string change)
- Add Alembic for database migrations (currently using `create_all` which doesn't handle schema changes)
- Deploy backend to AWS/GCP with managed Postgres (RDS/Cloud SQL)
- Deploy frontend to Vercel/Netlify
- Add connection pooling for concurrent users
- **Why it matters**: SQLite doesn't support concurrent writes — breaks with multiple users

### 4. Budget Goals & Tracking
- Let users set monthly spending limits per category
- Progress bars showing spend vs budget
- AI-powered budget recommendations based on historical patterns
- Push notifications when approaching limits
- **Why it matters**: The most-requested budgeting feature — the app currently shows what you spent, but doesn't help you plan

### 5. Recurring Transaction Detection
- AI-powered detection of subscription patterns (Netflix on the 15th, rent on the 1st)
- Calendar view of upcoming bills
- Alerts for price increases on subscriptions
- "Cancel this?" suggestions for unused subscriptions
- **Why it matters**: Subscriptions are a major source of wasteful spending

### 6. Export & Reporting
- CSV/PDF export of transactions and insights
- Monthly/yearly financial summary reports
- Tax-categorized expense reports
- **Why it matters**: Users need their data outside the app (accountants, tax filing)
